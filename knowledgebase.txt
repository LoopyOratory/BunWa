================================================================================
1. WHAT IS BUNWA
================================================================================

BunWa is a WhatsApp HTTP API server built on the Bun runtime with the Hono
web framework. It is a 1:1 API-compatible rewrite of WAHA (WhatsApp HTTP
API, https://waha.devlike.pro) that delivers the same functionality at
significantly lower resource usage — faster cold starts, less memory, no
Node.js.

Two WhatsApp engines are supported, selectable per session:

  - NOWEB (default) — uses Baileys (@whiskeysockets/baileys), a pure
    WebSocket implementation of the WhatsApp Web protocol (Signal protocol
    E2E encryption). Lightweight, no browser required, fastest option.

  - WEBJS — uses whatsapp-web.js + Puppeteer/Chrome. Runs an actual
    headless Chrome instance that drives web.whatsapp.com. Heavier, but
    gives full WhatsApp Web UI parity for edge-case features NOWEB doesn't
    cover yet.

Both engines can run simultaneously on the same server; each session picks
its own engine.

BunWa ships with:
  - A full WAHA-compatible REST API (sessions, messaging, groups, channels,
    labels, contacts, presence, profile, status/stories, webhooks).
  - A React 19 + shadcn/ui + Tailwind CSS dashboard (frontend/).
  - A WebSocket endpoint for real-time events (message received, ack,
    reactions, session status changes).
  - A Model Context Protocol (MCP) server so AI agents can drive WhatsApp
    directly (send messages, manage sessions, check numbers) via a
    standardized tool interface.
  - An audit logging system (SQLite-backed) tracking session lifecycle,
    message sends/failures, webhook delivery, and auth failures.
  - Interactive OpenAPI/Scalar API docs at /api-docs.

2. ARCHITECTURE
================================================================================

  Clients                     BunWa Server (Bun Runtime)
  ───────────                 ──────────────────────────────────────────
  HTTP Clients   ──────────▶  Hono REST API (WAHA-compatible endpoints)
  (curl, apps)                        │
                                       ├──▶ Session Manager (lifecycle,
                                       │     state, per-session config)
  AI Agents      ──────────▶  MCP Server (POST /mcp, stdio)             │
  (MCP clients)                       │                                │
                                       ├──▶ Tool Registry / Permission   │
                                       │     Checker (per-session policy)│
                                       ▼                                ▼
  Browser        ──────────▶  Dashboard (React) ──▶ WebSocket ──▶ Session
  (Dashboard UI)               + MCP Config API              Events

  Session Manager routes to:
    - NOWEB Engine (Baileys)  ──▶ WhatsApp WebSocket (Signal Protocol)
    - WEBJS Engine (Puppeteer)──▶ Chrome ──▶ WhatsApp Web

  Data Layer:
    - Database: SQLite (bun:sqlite, default) or PostgreSQL (via Knex)
    - Storage: Local filesystem or S3-compatible (AWS S3, MinIO, R2)
    - Session Index: .sessions-index.json (per-session config: webhooks,
      proxy, MCP policy, ignore rules)
    - Audit DB: SQLite, separate from session DB, in WAHA_STORAGE_DIR

Key design principle: every layer (database, storage, engine) is swappable
via environment variables alone — no code changes needed to go from SQLite
to PostgreSQL, or local storage to S3, or NOWEB to WEBJS.

The SessionManager (src/core/manager.core.ts) is the central orchestrator:
  - Loads/persists session configs (.sessions-index.json)
  - Starts/stops/restarts sessions, tracks status transitions
    (STOPPED → STARTING → SCAN_QR_CODE → WORKING, or → FAILED)
  - Merges WebSocket event streams per-session using RxJS, with a
    "wildcard" subscription mechanism (events2: DefaultMap<WAHAEvents,
    SwitchObservable<any>>) that re-subscribes whenever a session starts,
    so the Event Monitor sees events from sessions started *after* the
    WebSocket connection was already open (this was a real bug, fixed —
    see Section 8, item 1).
  - Emits audit log entries at every lifecycle transition (created,
    started, qr_generated, connected, disconnected, stopped, deleted,
    force_killed).

7. MCP SERVER (Model Context Protocol) — FULL DETAIL
================================================================================

BunWa exposes an MCP server so AI agents (Claude Desktop, Claude Code,
Cursor, custom MCP clients) can drive WhatsApp directly. Two transports:

  1. Streamable HTTP — POST /mcp. Stateless: a fresh McpServer + transport
     is built per request (no session map, no GET/DELETE reconnect
     handshake). Auth via X-Api-Key header or Authorization: Bearer.

  2. stdio — src/mcp/stdio.ts, spawned as a subprocess by the MCP client.
     Reads BUNWA_SESSION and BUNWA_MCP_KEY from its environment. Requires
     the BunWa source tree to be present locally (it runs `bun run
     src/mcp/stdio.ts` from the repo root) — this only works when the MCP
     client and the BunWa server/session-data are on the same machine; it
     is NOT viable against a remote/hosted BunWa deployment (Docker,
     Coolify) unless the stdio process has access to the same session
     data store.

23 tools total, in 3 categories:
  Session (6): SessionList (NOT session-scoped — lists every session's
    name+status server-wide), SessionGet, SessionStart, SessionStop,
    SessionRestart, SessionCheckNumber (last 5 are session-scoped)
  Messaging (16): MessageSendText, MessageSendImage, MessageSendFile,
    MessageSendVoice, MessageSendVideo, MessageSendLocation,
    MessageSendPoll, MessageSendContactVCard, MessageSendLinkPreview,
    MessageReply, MessageForward, MessageReact, MessageStar,
    MessageMarkRead, MessageStartTyping, MessageStopTyping (all
    session-scoped)
  Contact (1): ContactCheckNumber (session-scoped)

Auth model:
  - Global key: WAHA_API_KEY. Full admin access to all tools/sessions.
  - Per-session key: sk_mcp_<hex>, generated via dashboard (Session
    Settings → MCP → Generate Key) or POST /api/sessions/:session/mcp/
    generate-key. Only a SHA-256 hash is stored server-side
    (SessionConfig.mcp.apiKeyHash) — the plaintext key is shown exactly
    once at generation time and never persisted. Regenerating invalidates
    the previous key immediately.

Session scoping (auto-scoping):
  - When a request authenticates with a per-session key, the server looks
    up which session that key's hash belongs to (scopedSession).
  - For any tool marked sessionScoped: true, input.sessionId is FORCIBLY
    overwritten to scopedSession, regardless of what the caller passed —
    so a session-a key can never act on session-b's data even if it tries
    to pass sessionId: "session-b" in the tool call.
  - For non-sessionScoped tools (currently only SessionList), the server
    DENIES the call outright when authenticated with a scoped key. This
    was a real, confirmed security fix: SessionList used to be reachable
    by any per-session key and would leak the names/status of every
    other session on the server. Verified via live testing (see Section
    8, item 4). Global keys are unaffected — SessionList still works
    normally for admin access.

Per-session tool policies (SessionConfig.mcp):
  - enabled: boolean — master on/off switch for MCP on that session
  - allowedTools: string[] — allow-list mode; if set and non-empty, ONLY
    these tools/categories are permitted
  - deniedTools: string[] — deny-list; explicit deny always wins
  - destructiveOps: boolean — gate for tools marked destructive: true;
    must be explicitly true to permit irreversible operations
  - apiKeyHash: string — SHA-256 of the per-session key (see above)

Rate limiting: KeyRateLimiter, configurable via MCP_RATE_LIMIT_MAX /
  MCP_RATE_LIMIT_WINDOW_MS, keyed by the raw API key (or "anonymous").

Testing the MCP server:
  curl http://localhost:3000/api/mcp/tools | jq '.byCategory'
  bunx @modelcontextprotocol/inspector http://localhost:3000/mcp

Connection config formats (both shown with real values by the dashboard's
"Generate MCP Key" flow, each with a Copy button):

  stdio:
    {
      "mcpServers": {
        "bunwa-<session>": {
          "command": "bun",
          "args": ["run", "src/mcp/stdio.ts"],
          "env": { "BUNWA_SESSION": "<session>", "BUNWA_MCP_KEY": "sk_mcp_..." }
        }
      }
    }

  HTTP/SSE (works for Claude Desktop, Cursor, Windsurf, VS Code — all
  support the same url+headers remote-MCP config shape):
    {
      "mcpServers": {
        "bunwa": {
          "url": "http://<host>:<port>/mcp",
          "headers": { "X-Api-Key": "sk_mcp_..." }
        }
      }
    }

  NOTE: an earlier version of the dashboard's "legacy" Windsurf/VS Code
  guide referenced a package called "@openclaw/mcp-proxy" — this package
  does NOT exist on npm (confirmed 404) and was a documentation bug. It
  has been fixed to use the same url+headers format as the other clients.

8. HISTORY OF MAJOR FIXES (chronological, this development cycle)
================================================================================

1. Event Monitor wildcard WebSocket subscription fix (manager.core.ts)
   Bug: SessionManager.getSessionEvents() built a static merge() of event
   streams once, at WebSocket-connect time. Any session started AFTER
   the WS connection was already open was invisible to the Event
   Monitor — 0 events shown even though the session was actively running.
   Fix: events2: DefaultMap<WAHAEvents, SwitchObservable<any>> +
   refreshWildcardEvent()/refreshAllWildcardEvents(), called at every
   session lifecycle point (start, start-failure, stop). Verified live:
   went from 0 events received to full STARTING→WORKING→STOPPED coverage
   for a session started after the monitor page was already open.

2. Audit logging instrumentation (was fully built, zero call sites)
   AuditService (SQLite-backed, retention, query filters) existed but was
   never called anywhere except its own read endpoint. Instrumented:
   SessionManager (session_created/started/stopped/deleted/qr_generated/
   connected/disconnected/force_killed), api-key-auth.ts
   (api_key_auth_failed), chatting.routes.ts (message_sent/message_failed
   via a shared sendAndAudit() wrapper — later extended to also cover
   reaction and star, which had been missed), webhooks.routes.ts +
   webhook-delivery.ts (webhook_created/deleted/triggered/failed,
   including the SSRF-blocked case).

3. WhatsApp buttons/lists were completely non-functional
   Root cause (found via researching the actual WhatsApp binary protocol
   and a third-party button-helper npm package's source): Baileys'
   relayMessage() does not automatically attach the biz/interactive/
   native_flow/bot binary nodes that WhatsApp's server requires to
   actually RENDER interactive buttons/lists client-side — the protobuf
   interactiveMessage alone is not sufficient. A secondary bug: the code
   unconditionally wrapped every button message in viewOnceMessage, which
   no correct reference implementation does. Fix: noweb.buttons.ts now
   builds the binary nodes via buildButtonBinaryNodes() and passes them
   as additionalNodes to relayMessage(); the viewOnceMessage wrap was
   removed; sendList() in session.noweb.core.ts received the identical
   fix. Also found and removed a dead duplicate SendButtonsRequest DTO
   (only the one in chatting.buttons.dto.ts was actually imported/used).
   Verified: unit-tested node construction, live-tested against a real
   WhatsApp connection (correctly progressed to the "socket not
   authenticated" boundary, proving the code path itself was sound).
   NOTE: buttons/lists do not render inside a self-chat ("Message
   Yourself") — that's an inherent WhatsApp client-side limitation, not a
   bug — this caused a false-alarm bug report during development that was
   later self-corrected by the reporter.

4. MCP session-scoping leak (SessionList cross-session info disclosure)
   Every MCP tool except SessionList is sessionScoped: true. Session-
   scoped tools were already safe (auto-scoping forcibly overwrites
   sessionId), but SessionList had no scoping check at all — a key
   generated for one session could call SessionList and see the names
   and live status of every other session on the server. Verified live:
   a session-a-scoped key returned all sessions including session-b.
   Fixed by denying any non-sessionScoped tool call outright when
   authenticated via a per-session key. Verified: scoped key now gets
   PermissionDenied on SessionList, still works fine on its own
   SessionGet, global key's SessionList is unaffected.

5. WebSocket event envelope was missing entirely
   getSessionEvents() emitted raw payloads (WAMessage, WAMessageAckBody,
   WAMessageReaction) with no `.event` field, so the frontend's
   `if (!event) return` guard in the WS message handler always exited —
   live chat updates had literally never worked. Fixed by wrapping every
   emitted payload in { event, session, payload, timestamp }.

6. Chat UI: no live updates + sidebar flicker
   Two compounding bugs on top of #5: (a) once the envelope existed, the
   frontend needed a rewrite to incrementally patch messages/reactions/
   acks instead of doing nothing; (b) loadChats() unconditionally set
   loadingChats=true on every 5-second poll AND every WS-triggered
   refresh, blanking the entire sidebar to a loading skeleton constantly.
   Fixed via incremental WS handler + a "first load per session" ref flag
   that only shows the loading state once. Also added optimistic sending
   (a temp message appears instantly, reconciled with the real API
   response, rolled back on failure).

7. sendPoll used the wrong field name in the frontend
   Frontend sent { poll: { values: [...] } }; backend/DTO expects
   { poll: { options: [...] } }. Root-caused partly to a documentation
   bug: /api/sendPoll was defined TWICE in swagger.ts's object literal
   (a JS/TS duplicate-key situation — the earlier, detailed, CORRECT
   definition was silently shadowed by a later, vague, undocumented
   duplicate that never got fixed because nobody could see the good one
   was dead). Fixed the frontend field name and cleaned up the swagger
   duplicate. This surfaced a broader pattern (see #8).

8. Swagger/OpenAPI duplicate-path-key sweep (~37 total occurrences)
   swagger.ts is a single large object literal; JS/TS silently lets a
   later duplicate key overwrite an earlier one with no runtime error
   (though `tsc` DOES flag it as TS1117). Across two passes, ~37 paths
   were found duplicated: sendPoll, sendButtons, sendList, reaction,
   star, reply, sendLocation, sendContactVcard, sendLinkPreview,
   forwardMessage, startTyping, stopTyping, contacts/all, and the entire
   chats/groups/channels/labels/profile/presence groups. In several cases
   the winning (surviving) definition was actually WRONG relative to the
   real route handler — e.g. /api/reaction and /api/star were missing
   `chatId` as a required field even though the handler reads it; /api/
   reply documented a `reply_to` field that doesn't exist (the handler
   reads `messageId`). Two cases were worse: the winning /channels and
   /labels definitions were missing entire HTTP methods (POST create for
   both, DELETE for channels/{id}, GET for labels/chats/{chatId}) because
   only the dead duplicate had them — meaning real, working endpoints had
   ZERO live documentation. All duplicates removed, correct/missing
   details merged into the surviving definitions, verified by diffing
   the live-built OpenAPI JSON spec against the actual route handlers.

9. Docker/Coolify deployment fixes
   Missing bun.lock (was gitignored) broke reproducible installs — fixed
   .gitignore. Container crash-loop from a missing tsconfig.json in the
   production image (needed at runtime for Bun's decorator metadata) —
   added to the Dockerfile COPY step. Healthcheck was broken (used `curl`,
   not present in the slim image, against the wrong path) — switched to
   a `bun -e` fetch against /health.

10. Documentation bug: WHATSAPP_API_KEY vs WAHA_API_KEY
   CLAUDE.md (and, by extension, anyone following it) documented
   WHATSAPP_API_KEY as the auth variable. The code
   (config.service.ts:getApiKey()) only ever reads WAHA_API_KEY. Setting
   only WHATSAPP_API_KEY silently leaves the server in open/no-auth mode
   (since WAHA_API_KEY is then unset and WAHA_ALLOW_NO_AUTH defaults to
   allowing unauthenticated access) — confirmed live: with the wrong var
   set, every request succeeded regardless of the X-Api-Key header sent.
   Fixed the documentation; the code and .env.example were already
   correct.

9. KNOWN LIMITATIONS / NOT-YET-IMPLEMENTED
================================================================================

- Redis/BullMQ queue: environment variables exist and are reported by the
  Infrastructure dashboard page, but there is NO active queue backend
  wired up — the dependencies were removed. Setting WAHA_QUEUE_ENABLED
  etc. has no runtime effect.
- Per-session chat-filtering ignore toggles (ignore groups/channels/
  broadcast/status) shown in the dashboard's session settings are not
  yet applied by the engine — only the global WAHA_SESSION_CONFIG_IGNORE_*
  env vars take effect server-wide.
- MCP stdio transport requires the BunWa source tree and session data to
  be locally accessible to the process spawning it — not viable against
  a remote-only deployment unless session storage is shared/networked.
- Session deletion via the API (DELETE /api/sessions/:name) is a hard,
  immediate delete with no confirmation step, no soft-delete, and no
  undo — the auth state is gone from disk immediately. (This bit a real
  user during development: a stray UI test click deleted a live session;
  it had to be manually recreated and re-paired via QR.)
- WEBJS engine requires Chrome/Chromium to be present on the host via
  CHROME_PATH or PUPPETEER_EXECUTABLE_PATH; it fails with a clear error
  (not a raw module-not-found crash) if missing, but there's no bundled
  Chrome — you must provide one.

11. QUICK START (copy-paste)

  git clone https://github.com/LoopyOratory/BunWa.git bunwa
  cd bunwa
  PUPPETEER_SKIP_DOWNLOAD=true bun install
  cp .env.example .env
  # edit .env — at minimum set WAHA_API_KEY for production use
  bun run src/main.ts

    docker pull loopyoratory/bunwa:latest

    docker volume create bunwa-media
    docker run -d --name bunwa -p 3000:3000 \
      -v bunwa-sessions:/app/.sessions -v bunwa-media:/app/.media \
      -v $(pwd)/.env:/app/.env bunwa:latest

  Coolify: New Resource → Application → Build Pack: Dockerfile →
    Dockerfile Target: Dockerfile.coolify → port 3000 → add persistent
    volumes for /app/.sessions and /app/.media → set WAHA_API_KEY and
